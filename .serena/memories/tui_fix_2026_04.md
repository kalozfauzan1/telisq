# TUI Fix - April 2026

## Problem
Running `telisq run` did not display the TUI. The terminal would initialize but immediately exit without showing any UI.

## Root Causes
1. `App::run()` never called `start_tui()` to set up the terminal (raw mode, alternate screen)
2. `App` struct didn't have a `Terminal` instance
3. No `terminal.draw()` call in the event loop to actually render the UI
4. `Events::next()` returned `None` on timeout, causing the main loop to exit immediately
5. Orchestrator was not spawned as a background task, blocking the TUI event loop

## Fixes Applied

### cli/src/tui/app.rs
- Added `start_tui`/`stop_tui` imports from parent module
- `App::run()` now calls `start_tui()` to initialize terminal
- Added `main_loop()` method that takes `&mut Terminal` as parameter
- `main_loop()` calls `terminal.draw(|frame| self.render(frame))` each iteration
- Event loop continues on `None` events (timeout) instead of exiting
- `stop_tui()` called on cleanup

### cli/src/tui/events.rs
- `Events::next()` still returns `Option<Event>` but caller (main_loop) handles None by continuing
- Added handling for `TryRecvError::Disconnected`

### cli/src/commands/run.rs
- Changed `for task_spec in tasks` to `for task_spec in &tasks` to avoid moving tasks
- Added `.clone()` to `task_spec` in `add_task()` call
- Orchestrator now spawned as background task with `tokio::spawn()`
- TUI event receiver properly connected via `app.events.set_orchestrator_rx(event_rx)`
- Task specs loaded into app state for display

## Test Results
- All 75 tests pass
- Build compiles cleanly (only warnings, no errors)
